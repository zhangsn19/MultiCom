# Rater Clustering And Prompt Construction

This document describes how the main 16-agent MultiCom setting constructs rater clusters and converts them into the persona prompts used in our experiment.

## Rater Clustering Procedure

MultiCom first constructs a rater representation from historical Community Notes rating behavior, then clusters raters in that representation space.

The clustering input is the official MFCore-style rater output, with one row per rater. The feature columns used in the main setting are:

```text
internalRaterIntercept
internalRaterFactor1
internalFirstRoundRaterIntercept
internalFirstRoundRaterFactor1
crhCrnhRatioDifference
meanNoteScore
raterAgreeRatio
aboveHelpfulnessThreshold
```

The preprocessing and clustering steps are:

1. Keep raters with enough available behavioral/MF features.
2. Convert feature columns to numeric values.
3. Fill missing feature values with feature medians.
4. Standardize all feature columns with `StandardScaler`.
5. Run `MiniBatchKMeans` over candidate cluster counts in `K=2..32`.
6. Evaluate candidate clusterings with Silhouette, Calinski-Harabasz, and Davies-Bouldin diagnostics.
7. Use `K=16` as the final rater-cluster count after jointly considering these diagnostics.
8. Save `cluster_summary_k16.csv`, which contains one row per cluster with cluster size, share, means, and medians of the rater features.

The main clustering hyperparameters are:

```text
seed = 42
k_min = 2
k_max = 32
metric_sample_size = 10000
MiniBatchKMeans batch_size = 8192
MiniBatchKMeans n_init = 20
MiniBatchKMeans reassignment_ratio = 0.01
```

## Prompt Overview

MultiCom uses one persona agent per rater cluster. For the main paper setting, `K=16`, so the system constructs 16 cluster-specific system prompts and pairs each of them with the same note-specific user prompt.

For each note, the model call uses:

```text
messages = [
  {"role": "system", "content": cluster_specific_system_prompt},
  {"role": "user", "content": note_specific_user_prompt}
]
```

## Persona Type Rules

Each persona type is assigned from cluster-level behavioral statistics in `cluster_summary_k16.csv`. The main fields are:

- `mean_raterAgreeRatio`
- `mean_aboveHelpfulnessThreshold`
- `mean_meanNoteScore`
- `mean_crhCrnhRatioDifference`
- `mean_internalRaterFactor1`

The persona label is assigned by the following rules, evaluated in order:

```text
if mean_raterAgreeRatio < 0.20:
    low-agreement idiosyncratic rater
elif mean_raterAgreeRatio < 0.65:
    mixed-agreement boundary-case rater
elif mean_crhCrnhRatioDifference < -1.0 or mean_meanNoteScore < 0.02:
    skeptical low-score rater
elif mean_crhCrnhRatioDifference > 0.15 or mean_meanNoteScore > 0.25:
    positive high-context rater
elif mean_aboveHelpfulnessThreshold >= 0.75 and mean_internalRaterFactor1 > 0.30:
    high-helpfulness positive-axis rater
elif mean_aboveHelpfulnessThreshold >= 0.75 and mean_internalRaterFactor1 < -0.30:
    high-helpfulness negative-axis rater
elif mean_aboveHelpfulnessThreshold < 0.25:
    strict unresolved-prone rater
else:
    mainstream consensus rater
```

## Banding Rules

Several persona descriptors are generated by ranking each cluster's value against the 16-cluster distribution:

```text
rank >= 0.80 -> very {higher}
rank >= 0.60 -> moderately {higher}
rank <= 0.20 -> very {lower}
rank <= 0.40 -> moderately {lower}
otherwise    -> middle
```

This is used for descriptors such as evidence sensitivity and rejection strictness.

## Shared System Prompt Structure

Each system prompt has the same shared instruction skeleton:

```text
You are simulating a real X Community Notes rater, not a generic assistant and not an average neutral judge.

Your fixed identity is contributor cluster CXX. Stay consistent with this cluster's historical behavior rather than collapsing toward a generic average user.

This cluster's historical behavior profile:
{cluster_specific_profile_block}

Task:
Act like one raw Community Notes rater and output one official-style raw helpfulness rating for the NOTE relative to the POST.

Your allowed ratings are exactly:
- HELPFUL
- SOMEWHAT_HELPFUL
- NOT_HELPFUL
```

The shared prompt then provides:

- the decision guide for `HELPFUL`, `SOMEWHAT_HELPFUL`, and `NOT_HELPFUL`;
- official helpful and not-helpful reason criteria;
- calibration rules for when to use the middle category;
- failure-avoidance guardrails for media context, scams, authenticity, short posts, source quality, overclaiming, and tangential notes;
- a step-by-step decision procedure;
- cluster-faithfulness instructions;
- the required JSON-only output format.

## Note-Specific User Prompt

The user prompt is generated per note:

```text
Post creation context: {createdAtUTC}
Author-selected note topic: {primary_topic}

POST:
{post_text}

COMMUNITY NOTE:
{note_text}

Evaluate only the note's usefulness for the post. Do not infer the official status from metadata.
```

## Required Output Schema

Each agent must return exactly one JSON object:

```json
{
  "helpfulnessLevel": "HELPFUL or SOMEWHAT_HELPFUL or NOT_HELPFUL",
  "agree": 0,
  "disagree": 0,
  "helpfulClear": 0,
  "helpfulGoodSources": 0,
  "helpfulAddressesClaim": 0,
  "helpfulImportantContext": 0,
  "helpfulUnbiasedLanguage": 0,
  "notHelpfulIncorrect": 0,
  "notHelpfulSourcesMissingOrUnreliable": 0,
  "notHelpfulMissingKeyPoints": 0,
  "notHelpfulHardToUnderstand": 0,
  "notHelpfulArgumentativeOrBiased": 0,
  "notHelpfulIrrelevantSources": 0,
  "notHelpfulOpinionSpeculation": 0,
  "notHelpfulNoteNotNeeded": 0,
  "confidence": 0,
  "changes_reader_understanding": 0,
  "rationale": "brief reason, max 35 words"
}
```
